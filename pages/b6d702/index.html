# 提问：实现一个自己的instaceof

解析：这道题考察其实有两点，一点是考察原型，一点是考察对这些基础函数的实现（这点我猜测）。

答案：

function myInstanceof(target,origin){
  let proto = target.__proto__
  while(proto){
    if(proto == origin.prototype) return true;
    else return false;
    proto = proto.__proto__
  }
}

function minLen(arr){

    if(arr[i].length<len) len = arr[i].length;

# 提问：利用Promise模拟并发下载：一共有12个图片，同一时间并发数不超过4个

解析：这道题考察的是我们对promise的应用，同时也是考察我们对异步任务调度器概念

答案：

class downImgs{
  constructor(options){
    this.waitTask = [];
    this.doingTask = 0;
    this.max = options.max
  }
  addTasks(urls){
    urls.forEach((v,i)=>{
      this.addTask(v,i)
    })
  }
  addTask(url){
    if(this.doingTask < this.max){
      this.doingTask++
      this.downloadImg(url).then(()=>{  
        this.doingTask --;              
        if(this.waitTask.length>=1){
          this.addTask(this.waitTask.shift())
        }
      })
    }else{
      this.waitTask.push(url)
    }
  }
  downloadImg(url){
    return new Promise((resolve,reject)=>{
      const img = new Image();
      img.onload = function(e){
        resolve()
      }
      img.onerror = reject;
      img.src = url;
    })
  }
}

# 提问：vue3新特性

解析，闻这问题要么是想看看你有没有对新技术有所了解，要么就是他公司用了vue3

答案：

  1. 组合式api(setup)
  2. 响应式api
  3. 生命周期改动
  4. 多片段
  5. suspense（代替对组件v-if和v-else的产物，实验阶段）
  6. Teleport（将内容添加到指定元素下的标签）
  7. computed和watch发生变化，出现了watcheffect，并以第三配置项范生出了watchSynceffect/watchPostEffect
  8. 模版指令的改动（v-mode（可以传多个了，废弃自己定义传过来model）l、v-model自定义修饰符（modifiers））
  9. v-key绑定问题得到拓展，可以用在template上，v-if和v-for优先级改变
  10. 函数式组件的定义变化（因为有状态和无状态性能差异在vue3得到了很大修复，所以也不推荐函数式组件的使用了）
  11. is渲染组件的改变（在普通函数上不会直接渲染对应is的组件，除非给vue：前缀）
  12. 渲染api改动（渲染函数h变为全局，在setup使用要返回函数而不是h）
  13. slot的改动（2.6开始就改动了，slot和slot-scoped合并）
  14. 值穿透使用改变（/deep/变为::v-deep()，简写为:deep()）
  15. 新增全局样式（可以用两个style标签，一个带有scoped一个没带有。使用::v-global()伪类，简写:global()）
  16. 新增样式模块化
  17. 新增响应式css，不用再通过js或者给标签绑定的方式修改css了（在style标签中给v-bind，可以对setup中的数据进行绑定）
  18. 新增样式的插槽选择器（给插槽内容提供样式，直接写在父元素也可）
  19. provide/inject使用方式有了些变化
  20. 自定义指令发生了极大变化，出现了很多新的生命周期(以前只有bind，inserted/update/componentUPdated/unbind)
  21. data的接受者发生了改变，在vue2中能将data使用为对象那是不规范的写法，这一写法在vue3完全废除
  22. transition-group没有默认渲染根元素
  23. template必须有条件判断之类的，否则就是普通的template空标签
  24. 不会替换挂载元素
  25. 废除filter
  26. 从实例中完全移除了 $on、$off 和 $once 方法

实际面试当中不应该这样回答，回答个五六个就好了

# 提问：什么是跨域？跨域请求资源的方式有哪些？

解析，这个问题考核的知识点就如题目所说的是跨域，日常项目中还是经常使用到的

答案1：

跨域是在非同源情况所毁被限制dom/cookie/xmlhttp情况，而所谓的非同源指的是请求是的源和被请求的协议/ip/端口不一致。

知道了跨域是什么，那就要知道跨域是如何处理跨域这个东西。其实呢，我们发起跨域请求的时候是能顺利发出去的，响应也是能正常响应，这时候浏览器会做事有两件：一是服务端梅支持这个源的跨域，进行响应拦截；二是服务端有支持这个源的跨域，那么就正常响应。

答案2：

解决跨域请求资源的方式通常有三种：

  1. 前端使用proxy，将我们的源改成受支持的源
  2. 请求后端增加cors，支持我们的源
  3. jsonp，以前用的方案，但这方案缺点很麻烦，并且不支持post，也很难确定是否请求失败

# 提问：vite和webpack的区别

解析：这题考验的就是如题两个不同打包工具的特性区别，只能靠和理解，缺一不可。

答案：

  1. webpack在每次启动时会顺着入口文件去获取所有的模块将其孩换成成一个新的文件，这相当于要帮应用的全面代码都一次加载完毕才能启动项目，在项目越大时效率越差。vite则是在每次启动时对依赖的模块进行分类，分为依赖（通常指ode_modules中的代码和不会改变的纯JavaScript）和源码。vite使用go语言编写的esbuild对依赖模块进行预构建，比JavaScript要快上许多（go语言是一种新生的服务端语言，非常强大）。而则对源码模块则是基于ESM的方式，将部分的打包工作交给了浏览器处理，也因此vite只需要在浏览器请求源码的时候进行转换提供源码就可以了（按需）。
  2. webpack的更新速度比vite慢，因为webpack虽然做了各式各样的处理最后选择使用hmr进行热更新处理，但实际中发现这种热更新也是会随着应用变大而导致速度越来越慢。vite是在esm上执行，所以vite在编辑一个文件时能够精确的与最近的hmr边界失活（可指模块本身），这样一来无论应用大小hmr都能保持最快更新。简单说vite对hmr进行优化，并且还利用了http提高了页面的加载速度，源代码用协商缓存，依赖代码用强缓存（因为通常是不会变的代码了）

总结：

vite相比webpack项目的启动许多更快，热更新更好，但给生产环境使用时仍然需要打包（因为vite利用了很多http请求，如果不进行打包的话会有许多额外的网络往返），同时他虽然用esbuild提升了依赖模块，但他暂时还不能用来打包，因为还不够成熟

# 提问：浏览器存储方式的区别？

解析：这道题可能考验的更多是对浏览器存储方式历史变化，以及这些变化产物的区别

答案:

浏览器的存储方式有cookie，sessionStorage，localStorage

cookie：存储在cookie的数据是会在发起请求时自动一起带上，他的操作和设置都比较复杂， 通常需要我们进行封装或者直接用第三方封装好的cookie方法。然后因为cookir自动带在请求上，很容易被进行csrf攻击

seesionstorage：存储在seesionSttoage不会随请求发送，操作简单，但会在浏览器窗口关闭的时候消失

localStorage：同seesionStorage一样，但他不会随浏览器关闭而关闭，是持久缓存

总结：

cookie的安全隐患在seesionStorage和localStorage得到缓解

# 提问：组件化和模块化的区别是什么？

解析：组件化和模块化我们平时工作做肯定经常用到，但要是想把这个问题回答好很好。我觉得，问这个问题很可能就是听听你的见解？又或者喜欢你能给出他想要的回答？

答案：

当被问到这个问题可以蕾丝这么说。

组件的最初目的是代码的重用，他更偏向于提供一个相对独立功能的组件元素，可以说是有实物存在的。它通常位于代码层次的最下层，被其他代码依赖，且能过组合成一个全新的组件，能被嵌套和复用。

模块的定义则是包括函数，过程、子程序等等，可以说面向对象中的对象是模块，也可以说对象中的方法是模块。更简单一点说，模块是构成程序的最基本构件，通常对模块的封装会提供一个对外的接口。

说完了上面两则的特性，最后可以对此再做补充，因为可以回被接着问：

# 组件化能做到的事模块化也能做到，所以组件化到底有什么特别？

答案：

其实严格来说，组件化也是模块化的一种，它是模块化思维中衍生出来的一种新思维，也就是说组件化被包含于模块化。

了解以上之后，我们再说说两者在使用时怎么做直接的区分：

  • 目的性：组件化的目的是为了重用、解耦也就是要实现代码的高重用低耦合；模块化的目的是隔离和封装代码，也就是实现代码的高內聚，低耦合
  • 接口：组件化通常是直接使用，而模块化会对外提供借口进行使用
  • 代码架构定位：组件化是纵向的，模块化是横向的
  • 实现的成品：组件化通常会是一些ui组件库等等，而模块化就是一些业务的逻辑模块

然后，如果说完上面那些还不够，还需要做最后的总结就这样：

组件化是建立在模块化思想上的一种演化，组件化本就属于模块化的概念，他们都是对功能的拆分，但相比之下模块化不需要考虑重用行，而组件化则相反，还有就是一个偏向实体而一个偏向逻辑封装

# 提问：vue和react有什么区别？

解析：这道题考验的就是你对知识的探索精神，不然的话就是她公司两个都有用到，不然问这个问题其实是没有意义的

答案：

  1. 首先react更偏向设计模式mvc，vue更偏向mvvm
  2. 也就是说，react是单向数据流，vue是双向数据流
    1. 两个矿建都推崇组件化，但双方都组件又存在着一定的差异
  3. react推荐的组件化做法是jsx+inline style，简单说就是把html和css写在了js里面，而vue推荐的做法是单文件模版化，html/js/css在同一个文件中但又有做分离
  4. react的更新策略是自定下向下的全diff，也就是当状态改变时他会生成一个完整的虚拟dom，然后进行新旧对比，最后采用patch打补丁局部更新dom。而vue是通过跟踪每一个组件的依赖关系，当改变时使关联组件更新。
  5. 两者的diff算法思路基本一样，但实现上有略微差别（没特别去了解，但能知道思路时同一层级比较，不跨级）
  6. 数据驱动视图不一样，vue是mvvm很通俗易懂，react主要是依赖了setState，通过setState存储在pending队列，然后判断是否属于批量更新状态，是的话就加入组件队列，不是的就用updateCompent直接更新组件
