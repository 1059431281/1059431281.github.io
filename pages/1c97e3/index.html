浏览器兼容性篇

# 什么是浏览器兼容性？

随着w3c（万维网联盟标准，是一系列标准的集合）的推进，各类新浏览器内核出现，加上一部分保留的ie内核，各内核之间有着一定程度上的差异，导致在对相同页面处理不同。

这些问题有大有小，可以小到无法察觉，也可以打倒也没无法正常浏览，因此这一系列差异问题就统称为我们常说的浏览器兼容性。

# 为什么会有多内核的出现？

标准的推进是渐进性的，不是一下子就能定义出完美的标准，w3c标准也在日渐完善当中。

我们平时使用的浏览器绝非只有一种，而不同的浏览器厂商为了获取更多的市场份额，它们会研发出更符合w3c标准的内核，以至于有更好的兼容性。相对的，他们会提供一下浏览器独有的扩张功能，这对w3c标准也起到推进作用。

# 为什么多核会导致兼容性问题？

其实说直白一点，无非就两种原因：

1、 虽然目的一样，但在实现目的上的标准和思路不一样

2、相同内核，随着版本更新，一些旧属性会被丢弃，新属性增加，而使用旧内核的浏览器就会依然存在旧bug，以及对新属性不支持的情况出现

还有一种就是另类的，由程序员写出的不规范代码，导致不兼容性现象出现

# 浏览器和内核

目前主流的内核有四种：Webikit内核（实际上是唯一神）、Trident内核（除了最新版都不行了）、Gecko内核（靠火狐撑着）、Presto内核（快不行了）

浏览器和内核的对应关系：

**Chrome（谷歌）：**Blink（由Webkit泛生，是webkit的精简高效强化版。因为webkit本身的泛用性十分强大，可以说是适合各种各样平台的通用内核）

**Edge（替代ie的浏览器）😗*Blink

**Safair（苹果浏览器）：**webkit

**Firefox（火狐）：**Gecko

**Opera（欧朋）：**Blink（以前是presto）

**IE：**Trident

**QQ、搜狗、猎豹：**Treident（兼容模式下），webkit（高速模式下）

**百度、360：**双内核切换，ie和webkit都有

# 兼容性处理思路

  1. 渐进性推进：新旧浏览器双开以便于观察
  2. 降级处理：优先针对主流浏览器完成功能，再接着去测其他浏览器是否有问题

思路通常看人，但我更推荐第一种，因为那样虽然一开始麻烦一点，但不用像第二种完成后掉头回来看，一旦项目变得庞大会变得相同麻烦

# html兼容

这个能讲的不多，通常也不会遇到，主要要面对的问题就是相同标签写法，有的浏览器无法进行识别。

# 解决方案

在实际开发中，要面临还是html5新标签无法被识别的情况，这种无法被识别的标签通常会被解析成inline，因此我们只需要在初始化css给一些可能识别的标签加上block属性（但这不代表一些有特别功能的标签就会变得能用，只是能展示出来），不过这能起到的效果微乎其乎。

我们真正能解决这些问题，其一是可以通过自定义标签创建当前浏览器所没有得标签，其二就是直接使用第三方库html5shiv.js来解决，而这些必须建立在检测出浏览器所属版本确实没有对应标签的情况下。

# css兼容

比起html和js的兼容问题，css兼容才是我们浏览器兼容时的重中之重。

# Hack

全称为css hack，是针对不同浏览器/不同版本进行css编码过程的统称。出现的原因，就是不同浏览器对css的支持和解析呈现不同。

这东西现在用的比较少，它是一些特殊符合，只有对应的浏览器能够识别，通常分为三类：属性前缀法、选择器前缀法、IE条件注释法。

**注意：**在非必要的情况下不应当使用hack，使用它的前提是我们要考虑向下兼容，例如公司破天荒要我们兼容ie10以下的浏览器，那么可能就要使用一些hack了（最直接的原因就是css3在旧版本ie是没有的）。

# 浮动塌陷

这也是经常会遇到的问题，它发生的核心在于元素浮动后的高度不会被容器元素计算，解决方法通常有两种：

1、利用overflow:hidden将容器变为一个bfc，bfc中是会计算浮动元素高度

2、给浮动元素同级加入一个新元素，并给予清除浮动属性clear:both，通常使伪元素去这么做

# 兼容小技巧

上面提到的几个都是在工作中常见的情况，因为实际上随着ie和一些旧内核的淘汰，我们会遇到的问题不用再那样多样复杂，但这也不代表就不需要考虑，下面还是给出几个适用的。

  1. a标签要给样式时，最好设成inline-block或block
  2. 外部容器可以在最开始随意设置个适合的高度或者直接不设让其自适应，适配完毕后再进行高度调整
  3. 准备合适的初始化代码植入，省去很多代码，例如对li标签默认的list-style这些去除。
  4. 准备一个可以看见ie9的浏览器，通常ie9没问题后大部分浏览器都不会有问题，如果还有问题那就是跟firefox有关
  5. 工程化开发以来，我们可以通过第三方包对css进行兼容处理，但它仅限于对浏览器进行前缀处理，不能解决像上面所说的定位或塌陷问题（通常使用的包：postcss-loader,postcss-preset-env）

# addEventListener和attachEvent

addEventListener支持除ie9以外的大部分浏览器

attachEvent支持ie7，8，并且绑定时需要加上前缀on

# 解决思路

对函数进行重构，或者创建新的函数，包含两种方法进行判断使用，为保险起见可以加入第三种情况，也就是直接给dom元素绑定dom事件。

# classList和className

classList操作起来更加方便可靠，有add\remove\replace\toggle\container等方法，兼容性上，只要不是低于ie10的就没什么问题

className兼容性通吃，唯一的缺点就是会直接覆盖原元素所有样式

# 自定义属性

自定义属性有通过el.dataset.xx和el.getAttribute两种方式获取。

据说在firefox中无法使用dataset，但实测发现可以，或许是新版的firefox支持了（版本号：109.0 ），因此做出如下总结：

dataset可能存在不兼容旧版本火狐，以及经过检验发现不支持ie11以下浏览器

建议统一使用getAttribute，它仅不支持ie9以下版本。

# 块级变量

指的是let和const一类，ie11以下不支持。

# eval（想必没人会这么去用吧）

在ie中，可以使用它来获取指定id的元素，在其他浏览器不行

实测发现，在其他浏览器也可以，或许是新版本支持（没去进行旧版测验，暂保留）

例如：eval('test')可以获取id为test的元素

# 解决技巧

  1. 在封装组件之类的时候，可以试着去考虑全面一点
  2. 依然是使用第三方，它会帮我们补充好对应的兼容，例如块级变量一类，以及一些没提到的兼容问题，推荐的有：
    1. babel-loader,babel/core,babel/preset-env，缺点是不支持高级语法转换，如promise一类
    2. babel/polyfill，用的更多，会将所有兼容性代码引入，因此导致打包体积过大，优化相对麻烦
    3. core-js + 方法1，实现兼容。

翻译：

babel-loader，是让webpack能理解babel/core那些

babel/core，是babael的核心包，用来提供给babel-loader调用api

babel/preset-env，一个预设的插件集合，包含所有的es6转换为es5的规则（平时知道的browserlist，这里面就使用到它了）

babel/polyfill，preset-env虽然提供了规则，但不能为浏览器本身没有的功能进行添加，而polyfill能做到，将缺失的功能做js垫片，弥补低版本浏览器缺失的功能，缺点就是体积太大的问题。

因此，在使用polyfill的时候，我们不需要单独引入它，它会在使用地方主动注入。此外，polyfill通吃会把所有es6新功能添加垫片，因此我们需要在presets的选项里面配置usage，让其只对新功能做垫片。

# 总结

兼容性实际上发现的问题并不多，但往往会对此缺少总结，但去总结之后就会发现，这一类问题并不是多么高深
