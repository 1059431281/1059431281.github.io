keep-alive实现原理

# keep-alive的实现原理

本片文章就来简单记录下去年被面试官狠狠揉捏的keep-alive原理，那么...开始！

# keep-alive是什么？

所谓的keep-alive也就是我们vue中常用的缓存标签，用来缓存组件的状态，并新增了有关actived的生命周期。

# 一般会在什么时候使用呢？

当我们从a页面跳到b页面，然后又从b页面跳回来的时候希望仍旧是离开a页面时的状态。实际一点的例子的话，就比如我们平时使用百度贴吧的时候，我们点进别人的帖子又跳回来的时候，保留着我们刚刚浏览到的位置和数据。

# keep-alive标签的props有哪些？

keep-alive的props主要是用来设置缓存条件，也就是哪些要缓存哪些不要缓存以及最大缓存数

  1. inlclude： 只有名称匹配的组件会被缓存（vue文件里面的name）,可以使用正则表达/逗号划分字符串/数组
  2. exclude：与之不名称匹配的组件才会被缓存,可以使用正则表达/逗号划分字符串/数组
  3. max：最大可以缓存多少组件实例（lru算法会讲）

# pruneCache

/* 修正cache,这里的代码被我改动了一下，因为不同版本vue实现由有点不一样 */
function pruneCache (proxy, filter: Function) {
  let cache = porxy.cache
  let current = proxy._vnode
  for (const key in cache) {
    /* 取出cache中的vnode */
    const cachedNode: ?VNode = cache[key]
    
    if (cachedNode) {
      const name: ?string = getComponentName(cachedNode.componentOptions)
      /* name不符合filter条件的，同时不是目前渲染的vnode时，销毁vnode对应的组件实例（Vue实例），并从cache中移除 */
      if (name && !filter(name)) {
        if (cachedNode !== current) {
          pruneCacheEntry(cachedNode)
        }
        cache[key] = null
      }
    }
  }
}

# pruneCacheEntry

function pruneCacheEntry (vnode: ?VNode) {
  if (vnode) {
    vnode.componentInstance.$destroy()
  }
}

总结一下就是，include和exclude改变后会遍历cache中的所有项，如果不符合filter指定的规则的话，则会执行pruneCacheEntry。pruneCacheEntry则会调用组件实例的$destroy方法来将组件销毁。

# Matches

/* 检测name是否匹配 */
function matches (pattern: string | RegExp, name: string): boolean {
  if (typeof pattern === 'string') {
    /* 字符串情况，如a,b,c */
    return pattern.split(',').indexOf(name) > -1
  } else if (isRegExp(pattern)) {
    /* 正则 */
    return pattern.test(name)
  }
  /* istanbul ignore next */
  return false
}

# 最后就是核心的render渲染函数了(这里直接转载别人写的了，因为写的真的好)

  // 自定义render函数
  render () {
    /*
    * 获取第一个子元素的 vnode
    * 由于我们也是在 <keep-alive> 标签内部写 DOM，所以可以先获取到它的默认插槽，然后再获取到它的第一个子节点。<keep-alive> 只处理第一个子元素，所以一般和它搭配使用
    * 的有 component 动态组件或者是 router-view，这点要牢记。
    */
    const slot = this.$slots.default
    const vnode: VNode = getFirstComponentChild(slot)

    const componentOptions: ?VNodeComponentOptions = vnode && vnode.componentOptions
    if (componentOptions) {
      // check pattern
      // 判断当前组件名称和 include、exclude 的关系：
      const name: ?string = getComponentName(componentOptions)
      const { include, exclude } = this

      // matches就是做匹配，分别处理了数组、字符串、正则表达式的情况
      // 组件名如果满足了配置 include 且不匹配或者是配置了 exclude 且匹配，那么就直接返回这个组件的 vnode，否则的话走下一步缓存：
      if (
        // not included
        (include && (!name || !matches(include, name))) ||
        // excluded
        (exclude && name && matches(exclude, name))
      ) {
        return vnode
      }

      const { cache, keys } = this
      const key: ?string = vnode.key == null
        // same constructor may get registered as different local components
        // so cid alone is not enough (#3269)
        ? componentOptions.Ctor.cid + (componentOptions.tag ? `::${componentOptions.tag}` : '')
        : vnode.key
      // 如果命中缓存，则直接从缓存中拿 vnode 的组件实例，并且重新调整了 key 的顺序放在了最后一个
      if (cache[key]) {
        vnode.componentInstance = cache[key].componentInstance
        // make current key freshest
        // 使用 LRU 缓存策略，把key移除，同时加在最后面
        remove(keys, key)
        keys.push(key)
      } else {
        // 没有命中缓存，则把 vnode设置进缓存
        cache[key] = vnode
        keys.push(key)
        // prune oldest entry
        // 配置了max 并且缓存的长度超过了 this.max，则要从缓存中删除第一个
        if (this.max && keys.length > parseInt(this.max)) {
          // 除了从缓存中删除外，还要判断如果要删除的缓存并的组件 tag 不是当前渲染组件 tag，也执行删除缓存的组件实例的 $destroy 方法。
          pruneCacheEntry(cache, keys[0], keys, this._vnode)
        }
      }
      // keepAlive标记位
      vnode.data.keepAlive = true
    }
    return vnode || (slot && slot[0])
  }
}

看完以上代码后，再继续一下缓存步骤总结：

  1. 判断当前组件是否需要缓存，需要就缓存去cache对象，不需要就直接返回实例
  2. 要渲染某个组件之前，会根据组件id和缓存key去查找是否缓存过该对象，如果缓存过就直接取出进行渲染，并将其挪到cache对象首位
  3. 要渲染某个组件之前.....没发现有缓存过该对象，那么就先检查当前cache对象长度是否超过了max，没超过就直接加入缓存，如果超过就删除cache尾部的缓存记录，往头部加入缓存

# abstract（抽象组件）

我们在源码内能看见，我们在created的时候给饿了abstract为true，表明是抽象组件。

既然是抽象组件，那么它就不会被渲染成dom。

# 首次渲染和缓存渲染

首次渲染的时候，会正常的去总vue的生命周期，而进行缓存渲染的时候则会通过判断vnode.componentInstance和vnode.data.keepAlive不执行created/moutned这些钩子函数，而是会走actived钩子函数

# 缓存策略（LRU）

这个就是我们在置换cache时所用的核心算法思路，实现的判断大致如下：

  1. 新数据会被插入到链表头
  2. 缓存命中时，将命中的数据推移到头部，其他数据往后退一步
  3. 链表满时，丢弃链表尾部数据
