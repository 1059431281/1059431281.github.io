设计模式

本文是纯唤醒记忆用，不适合零基础，示例代码js和ts都有。

# 前言

文章分为两部分，一种是我们日常就经常在使用的设计模式，一种是我们应该会使用并且就算使用也没意识的设计模式。

# 组件模式

组件模式是我们日常大量使用的设计模式，他的出发点就是给每个组件独立的上下文，组件之间的隔离型强，复用性高。通常来说，在不考虑全局变量的影响下，组件之间不应有潜在交流。

基本介绍完，接下来就是它存在的一些缺点。

比方说，我们的page有一个顶部标题，我们将它独立出去组件，这样做的目的便违背了我们组件化复用的初衷，并且破坏了我们原有组件的完整性，就连传递props也要多次。

为此，组合模式的存在是必要的。

# 使用场景：

复用、解耦

# 组合模式

组合模式，其实说到底就是把分离出去的组件再进行组装，形成一个新的组件。这样一来，这个新生成的组件，里面构成它的组件之间是运行互相传递的。

**小心误解：**这里可能会产生一种误解，那就是组合模式难道是一种组件潜逃组件的高阶组件，其实不是，理解成mixins才更加准确。

总结特性：

  1. 组件模式拥有独立上下文，而组件和组件之间组合成新的组件需要上下文传递，可以将组合模式理解为是组件模式的一部分。简而言之，若干个组合体组成了一个完整组件，组合体之间共享上下文，不需要额外传递，至于内聚，组合体本身就实现了相关逻辑内聚。
  2. 组合体之间不能有相同的内部成员。
  3. 组件有实例，需要命名标识，组合体没有实例，不能命名标识。

# 使用场景：

组件之间需共享数据多

# 单例模式

单例模式的核心是确保一个类只有一个实例，并且提供全局访问api。（vuex/vue-router这些都是很好的例子）

单例模式有很多种实现方式，具体用哪种由自己决定。

# 简单实现

# class方式

class Singleton {
  name: string;
  static instance: any;
  constructor(name:string) {
    this.name = name;
  }
  static getInstance(name:string) {
    if(!this.instance) {
      this.instance = new Singleton(name);
    }
    return this.instance;
  }
}

# 可new形式

上面如果我们不是被告知了根本不知道他是单例，也不会那么去用，所以我们要将其变成可new的，这样更通俗易懂。

这个实现起来不难，只需要在全局定义一个变量，然后在要进行new的函数里面用类似我们上面的判断就好。

但问题又来了，在全局定义变量？光从变量污染这点来看就不好了。

还有一点，我们上面在实现时，并没有遵守设计模式的单一原则，即一个类实现一个功能，所以我们需要把基础功能和创建单例功能分开。

基于以上，就需要代理了。

interface Test{
  (name:string):void;
}

var createSingleton = (function(){
  let instance:Test;
  return function(Singleton:Test,name:string){
    if(!instance) {
      return instance = new (Singleton as any)(name)
    }
    return instance;
  }
})()

var test:Test = function(this:any,name:string){
  this.name = name;
}

let a = new (createSingleton as any)(test,"a")
let b = new (createSingleton as any)(test,"b")

# 惰性

简单解释一下，就是在需要时才去进行实例，并且只实例那么一次，让其不至于被浪费。

具体来讲，假如说一个弹窗，是要v-show还是v-if呢（这里沿用vue的概念，其实就display和重新创建元素的区别）？答案是两者配合着来，这才单例模式的核心。

什么意思呢？假如这个弹窗一直没被用，那么单v-show的话他虽然是none但却还是存在在页面上，很明显浪费了。那么v-if呢？如果弹窗一直被反复使用，反反复复的渲染卸载更加浪费。为此，单例的好处就来了，在第一次去渲染元素，后续再用v-show去隐藏和显示。

下面，给出相对完整的js写法代码。

var getSingle = function (fn) {
  var result
  return function () {
    return result || (result = fn.apply(this,arguments))
  }
}

let testa = function(name){
  let div = document.createElement('div')
  return div;
}
let createSingleton = getSingle(testa)

# 发布-订阅模式

简单解释，基于一个事件（主题）通道，将会被通知的对象，也就是订阅者可以通过自定义事件去定义主题，发布者则通过发布主题的方式通知各个订阅了对应主题的订阅者。

const PubSub = function(this:any){
  this.events = {};
  this.subscribe = function(name:string,cb:(...arg:any)=>any){
    if(!this.events[name]){
      this.events[name] = [];
    }
    this.events[name].push(cb)
  }
  this.publish = function(name:string,...arg: any){
    if(this.events[name]){
      this.events[name].forEach((e:(...arg:any)=>any)=>{
        e(...arg)
      })
    }
  }
}

# 策略模式

理解这个也很容易，我们平时会在函数内使用大量的if-else，其实在非必要时可以用策略模式代替。

为什么呢？因为大量的if-else显得代码臃肿，并且新增的时候又需要在函数里面重新写，违背了开闭原则。

interface Types {
  a:(name:string)=>void
  b:(name:string)=>void
  c:(name:string)=>void
  d:(name:string)=>void
}

let types:Types = {
  a:(name)=>{
    console.log(name+"a")
  },
  b:(name)=>{
    console.log(name+"b")
  },
  c:(name)=>{
    console.log(name+"c")
  },
  d:(name)=>{
    console.log(name+"d")
  }
}

interface MySelf {
  (type:keyof Types,name:string):any
}

var myself:MySelf = function(type:keyof Types,name:string):any{
  return types[type](name)
}

myself("a","小明")
myself("b","小红")
myself("c","小绿")
myself("d","小蓝")

# 命令模式

这个模式字面意思来讲的话，就是命令别人帮忙做事。然后简单理解一下，其实就是消除调用者和接收者之间的耦合，解译一下就是，给两者之间加入一个代理者。而代理者的好处，除了解耦之外就是记录回撤功能。

这里以服务员下单为例子。

class Proxy {
  constructor(proxys) {
    this.proxys = proxys;
    this.oldCommands = []
  }
  execute(type){
    let id = this.proxys[type].execute().id
    this.oldCommands.push(id)
  }
  unpop(){
    this.oldCommands.pop();
  }
}

var menu = {
  variety1:{
    execute:function(){
      let id = `data${+Date.now()}`
      return {
        id
      }
    }
  },
  variety2:{
    execute:function(){
      let id = `data${+Date.now()}`
      return {
        id
      }
    }
  },
  variety3:{
    execute:function(){
      let id = `data${+Date.now()}`
      return {
        id
      }
    }
  }
}
const waiter = new Proxy(menu);
waiter.execute("variety1")
waiter.execute("variety2")
waiter.execute("variety3")
waiter.unpop()
console.log(waiter.oldCommands) // 只剩下variety1和variety2

# 责任链模式

责任链模式的初衷，简单就是将请求发送者和接收者进行解耦，让多个对象都有能力接收请求，然后将对象连成一条链，沿着这条链传递请求。

可能这样说还不好理解，看一个实际例子：

  let res = data + 3
  d(res)

  let res = data + 4
  console.log(res)

责任链写法

function Chain(fn){
  this.fn = fn;
  this.next = null;
  this.setNext=(nextChain)=>{
    this.next = nextChain;
    return nextChain;

  }
  this.run=(...arg)=>{
    let ret = this.fn(arg);
    if(this.next){
      this.next.run(ret);
    }
  }
}

function a(data){
  let res = data + 1
  console.log(res)
  return res;
}
const chainA = new Chain(a)
function b(data){
  let res = data + 2;
  console.log(res)
  return res;
}
const chainB = new Chain(b)
function c(data){
  let res = data + 3
  console.log(res)
  return res;
}
const chainC = new Chain(c)
function d(data){
  let res = data + 4
  console.log(res)
  return res;
}
const chainD = new Chain(d)
chainA.setNext(chainB)
chainA.run("一系列操作后")
chainB.setNext(chainC).setNext(chainD)
chainB.run("一系列操作后")

const door = function(material="木料"){

const red = function(thing){

const test = new red(Door)

# 代理模式

代理模式，就是两个真实对象之间的沟通需要一个中间桥梁。最好的理解思路就是，代购！

a买b的东西，但是自己基于各种原因没办法买，所以要委托c去买。

下面代码示例由《JavaScript 设计模式与开发实践》得来

const domImage = (function(){
  let imgEle = document.createElement("img");
  document.body.appendChild(imgEle);
  return {
    setSrc: function(src){
      imgEle.src = src;
    }
  }
})()

const proxyImage = (function(){
  let img = new Image();
  img.onload = function(){
    domImage.setSrc(this.src)
  };
  return {
    setSrc: function(src){
      domImage.setSrc('xxx');
      img.src = src;
    }
  }
})()

proxyImage.setSrc("xxx")
