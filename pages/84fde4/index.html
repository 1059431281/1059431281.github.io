Hook是什么

# 前言

在写这篇文章之前我对hooks的概念也是很混乱，可以说平时有可能会用到，但我根本不知道他是不是hook，有什么好处，该怎么理解这个东西，所以我就花了不少时间通过各种渠道进行了学习总结

# hooks的定义

想理解hooks什么的话，就来看hooks是什么。

说到hooks我们的固有印象是联想到react，但他其实不是react独有的，而是整个前端界的一种专业术语，但要说到底是谁最先提出这个概念我认为是react。

那么hooks到底是什么？有没有直接一点说明的语句，有的，它们是这么定义的。

系统运行到某一时期时，会调用被注册到该时机的回调函数

什么意思呢？其实指的就是在某一时候会用的函数。

再通俗一点的话，就拿我们浏览器的onload和addeventListener来举例，这两个方法所绑定的回调函数就是我们所说的hook

再hook16.x之前hooks更多的被说是生命周期，而在这之后的hooks会根据不同的场景有不同的意思。

打个比方，在react中它是指use开头的函数方法，用来规避calss式写法，提供函数式编程所需要的生命周期/状态管理/逻辑服用等等所有class式写法会有的开发能力。

# 总结：react中提供给函数式组件的开发工作能力的那些函数式hooks

那么在vue里面呢？

hooks的概念是在vue3中才有的，它是在组合式api中以use开头的那一系列提供组件复用，状态管理 等等开发能力的方法。但是呢，通过其他资料的查阅，除了use开头的，组合式api中生命周期/ref等等其实也可以说是属于hooks，但这种说法不一定准确，因为不符合hooks规范。概括一下就是组合式api就是vue的hooks。

# 两者的共性：

react中，提供给函数式组件，并且只能给函数式组件用的方法们是hooks

vue中，提供给setup使用，并且只能在setup中使用的方法是hooks

# hooks的命名规范

因为react是最早（好像）提出hooks这个概念的，所以他们在定义的时候提出一个假设思想。

**一个假设：**假设然后以use开头后跟一个大写字母开头的函数是hook

为了支持react的这一假设，他们还提供了一个eslint插件来强制hook规范。

因为在react有这么一个总结性的假设，所以我在上面的组合式api中提到的生命周期/ref这种才说不符合规范。

这明明规范虽然不是强制的但却是一种很好的约定，否则大家都胡乱取名的话我们也不会去区分。

# hooks的使用规则

react对hooks的使用有来个限制：

  1. 只在最顶层使用hook（不要循环，调教或嵌套函数中使用hook）
  2. 只在react函数中调用hook

同样的，在eslint也有相应的插件。

# 命名规范和使用规则总结

通过上面两点，我们暂且就以react的明明规范也算是vue的hook命名规范，而使用规则在vue中也同样可以沿用，只在setup函数中调用hook

# Hooks有什么好处？

# 复用性

用过react或vue的应该都了解mixin这个东西吧，它是是我们实现状态逻辑复用的方法。

既然用过mixins的话，那么自然应该清除它的缺点：

  1. 在混入多个文件或者采用嵌套混入的时候，我们难以确定方法和属性属于哪个文件，也就是难以追溯
  2. 在混入多个文件的时候，如果混入文件中有相同的属性火方法会出现同名现象，后混入的文件会覆盖先混入文件的同名
  3. 假如我们重复使用mixins的内容我们要怎么办，难道要重新引入两次mixins？很显然是没办法那么做的，为此我们要做的处理是相当麻烦的，要去动态生成mixin，无论可读性还是复杂都很麻烦

基于以上明显的缺点，hook能几乎完美解决这一问题。

为什么解决了？我们来看一下它的使用：

import { useName } from "xx"
const { name, setName} = useName()
通过这样的方式，我们能清晰的看见这些属性是由谁产生的
const { name:Name1,setName:setName1} = useName();
因为是函数返回值，所以我们可以直接通过给变量别名，就能够重复使用了,同时解决了覆盖存在的问题

# 代码高内聚/低耦合（代码组合）

react我写的不多不是清除，但在vue2中我们的代码都是分在不同的地方进行编写，data/methods/computed和各个生命周期，但在vue3中提倡我们将这些写在一起，实现这一目标的也是依靠hooks

我们将分散的代码集中写在了setup中，利用hooks提供的方法就能做到这一点，显而易见提高了代码的内聚且大大降低了各生命周期的耦合。

# this的绑定问题

在react中的class写法最为明显，我们经常需要去执行bind(this)，这使得我们代码变得难看复杂，万一忘记bind一下就会报错。

至于vue的话，虽然不明显，但它存在着这样的问题，如果我们用肩头函数去写computed/mtdhos那些的话this会变成undfined

# 渐进式

这点算不上是hooks独有的优点，因为只是react和vue支持着hooks写法同时也支持着旧写法，像react还支持class类写法，vue还支持vue2的写法

# 怼人(面试官)用

你可以用上述的方式进行回答，然后如果要总结的话可以这么说：

hooks是定义在某个运行时刻会被触发的回调函数，并且根据不同场景会有不同的含义，通常以use+大写开头的函数名组合为hook名，在vue和react有各自的使用限制，提高了状态的复用性，实现代码高内聚
