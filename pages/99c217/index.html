ts学习总结——实际篇

总结完了基础篇一些点，接下来再总结一下实战中要注意点的知识点。

# 文件分配

在实际开发中，比较代表性的文件夹分配是：

# types

不放在src下，而是跟src同级，因为他是一个存储用来保证ts不会检测出错的文件，没必要放src里面。

该文件夹下存放的是xx.d.ts文件，它的作用自然不需要多说，一种是用来加强第三方引入没加type描述，一种就是用来定义我们在项目中会用的类型。

# 例

export interface Result<T = any> {
  code: number;
  type: 'success' | 'error' | 'warning';
  message: string;
  result: T;
}

# import和import type

两者最大的区别，就是import type导入的会被定位变量，只会在ts检测时起到作用，不会在运行时时存在。

因此，我们在实际中会大量使用到import type，但有一点要注意。作为刚入门的人来说可能会把枚举误导用import type导入，这样是错误的。

# interface和type

在实际开发中，如果弄不清要用哪个的时候，优先使用interface。

# 繁琐问题

使用ts开发的时候，很容易陷入为了实现ts的检测而给每个变量都去定义类型的问题，小至普通变量，大致函数参数，返回值一类。

其实呢，在开发中并不需要给所有变量都加上检测。别忘了，我们用ts的初衷时为了更好的维护和可读，但这随之加来的是工作量的猛涨。不仅如此，可能还会出现本末倒置的现象，想想一些一看就懂的临时变量什么的，我们都给定义，是不是无用功？当然，这里并非是说临时变量就不需要创建，要视情况而定。而且，别忘了ts还有自推导特性。

自推到特性，就是它可以根据赋值一方推导形成自己的类型。

interface hh {
  name: string;
  age: number;
  sex?: number
}

let a:hh ={
  name:"123",
  age:123,
}

let b =  {...a};

b.sex = 18
b.sex2= 19 // error

# 别名的使用

在ts项目中，使用alias的时候，不单单是在脚手架配置文件里面给予就好，还需要在tsconfig.json使用path将别名定义好路径，否则虽然不影响运行，但还是会出现错误警告。

# 搞清楚类型的存在位置

我们在实际开发工作中，如非必要不要在我们包括逻辑处理等文件（包括我们常见的vue文件）中去定义类型，应该创建独立的文件，在全局的也好，在当前文件夹也好，总之不要直接在逻辑里面定义，要使用时通过导入的时候就好。
